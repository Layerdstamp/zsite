# Step 3 Project Organization Plan

## Purpose

Turn the current idea into a manageable project structure so future work is not reactive, messy, or out of order.

This document is not about design or coding. It is about:

- project logic
- workstream separation
- sequencing
- scope control
- content organization
- decision timing

## Core Project Definition

This project is one personal brand ecosystem under one domain.

It has three connected experiences:

1. Home
- a minimal link-tree style landing page
- fast access to socials
- entry point into the deeper site branches

2. Art
- a portfolio branch
- rebuilt from the existing art website
- not treated like a redirect or external destination

3. A Look Into My Life
- a lifestyle and editorial branch
- built as a softer personal world with articles, galleries, and future premium content

## Project Rules

These rules should control planning and prevent the repo from becoming jumbled again.

1. One domain, one ecosystem
- no major branch should feel like a disconnected side project

2. Homepage stays minimal
- the homepage is a portal, not a content dump

3. Art and Life are the deep-content branches
- dense written content and large media sets belong there, not on the landing page

4. Preserve before simplifying
- existing images and descriptions from the current sites are intended to be carried over before any content reduction decisions are made

5. Separate planning from build execution
- naming, content grouping, migration mapping, and monetization strategy should be organized before visual execution starts

6. Private content is a product layer, not a random folder
- premium content belongs inside A Look Into My Life as part of a membership or gated-content strategy

## Primary Workstreams

The project should be managed as six separate workstreams.

### 1. Brand Structure

Purpose:
- define what each branch is responsible for
- prevent overlapping content and confused navigation

Already mostly decided:
- Home = portal
- Art = portfolio
- A Look Into My Life = personal editorial world

Still to lock later:
- final public-facing route names
- final top-level navigation labels

### 2. Content Migration

Purpose:
- move all existing website material into the new ecosystem without losing assets or copy

This is a major workstream, not a side task.

Content sources include:
- current Bio Sites landing content
- current Wix art website
- existing repo planning and category notes
- future personal article and gallery material

Main requirement:
- every image and description from the current live sites should be inventoried before migration decisions are finalized

### 3. Content Architecture

Purpose:
- define where each type of content belongs
- prevent the lifestyle branch from becoming a random bucket

This workstream should answer:
- what is a page
- what is an article
- what is a gallery
- what is a category landing page
- what is premium-only
- what stays public

### 4. Monetization Planning

Purpose:
- define how premium content is positioned before it is designed or built

This should stay separate from basic sitemap work.

Needs later decisions on:
- members-only subscription
- one-time paid drops
- hybrid access model
- what content belongs behind the gate
- what remains public teaser material

### 5. Domain and Hosting Strategy

Purpose:
- decide where the finished site lives and how the branch structure will be published

Later decisions needed:
- exact domain
- GitHub Pages vs Cloudflare direction
- whether premium features require a platform change or external service

### 6. Visual Direction

Purpose:
- define the aesthetic system only after structure and content logic are stable

This workstream is intentionally later.

Do not decide yet:
- final visual theme
- animation style
- final banner image
- premium experience styling

## Recommended Project Sequence

The project should move in this order.

### Phase 1. Structural Planning

Goal:
- lock the site map and branch responsibilities

Status:
- mostly complete through Step 1 and Step 2

Output:
- stable project logic

### Phase 2. Content Inventory

Goal:
- identify every piece of content that exists now

Needed outputs:
- complete image inventory
- complete page inventory
- complete description / copy inventory
- source-to-destination mapping

Reason:
- without this phase, future build decisions will be based on incomplete memory instead of actual source material

### Phase 3. Content Mapping

Goal:
- assign each source asset and each written section to its final destination in the new site

This phase should produce:
- Art migration map
- A Look Into My Life migration map
- public vs premium split
- public naming decisions for sections and pages

### Phase 4. Product Decisions

Goal:
- decide the role of premium content and any gated experiences

This should answer:
- what is premium
- how it is accessed
- whether it is present at launch or later

### Phase 5. Visual Planning

Goal:
- choose visual language only after the structural and content logic is stable

Reason:
- aesthetics should support the architecture, not hide weak organization

### Phase 6. Build Execution

Goal:
- implement the site once the content and branch logic are stable

## Branch-Level Planning Priorities

### Home priority

Need to keep clear:
- name
- banner structure
- social links
- entry to Art
- entry to A Look Into My Life

Need to avoid:
- large galleries
- long writing
- too many buttons
- category overload

### Art priority

Need to define later:
- exact sections from the current art site
- featured work structure
- collection grouping logic
- About Me and My Skills placement

Need to avoid:
- mixing lifestyle content into the portfolio branch

### A Look Into My Life priority

Need to define more deeply:
- article categories
- gallery categories
- public vs premium split
- how soft-membership feeling is expressed structurally

Need to avoid:
- turning this branch into an unsorted archive of personal content

## A Look Into My Life Content Model

This branch needs stronger internal rules than the others.

Recommended content types:

1. Category landing pages
- Spirituality
- Womanhood
- Style
- Travel
- See More Of Me
- Private Gallery

2. Article pages
- longer written content
- advice, reflections, personal voice pieces

3. Gallery pages
- grouped image sets with optional short intros

4. Hybrid pages
- writing plus image-led storytelling

5. Premium pages
- locked material reserved for future paid access

## Current Organizational Risks

These are the risks most likely to create confusion later.

1. Route names may be treated as final too early
- keep them provisional until content mapping is done

2. The premium concept may stay vague for too long
- if this is not defined before build, it will distort the structure later

3. The art-site migration may be underestimated
- portfolio rebuilds often contain more page-level detail than expected

4. The lifestyle branch may grow too wide
- without clear content types, it can become a catch-all section

5. Visual planning may start too early
- this risks beautifying an unfinalized structure

## What The Project Needs Next

The next planning layer should not be design.

The next planning layer should be a migration and content map that answers:

- what exact source pages exist
- what exact source assets exist
- where each one belongs in the new site
- which content is public
- which content is premium
- what pages are required for launch versus later phases

## Suggested Immediate Planning Deliverables

1. Step 4 Content Inventory Plan
- list every source to scrape or extract
- define how assets and descriptions will be tracked

2. Step 5 Content Migration Map
- assign source material to final destination pages

3. Step 6 Launch Scope Plan
- define what is required for version one and what is phase two

## Practical Working Rule From Here On

For the rest of planning, every new idea should be sorted into one of five buckets before anything else:

- Home
- Art
- A Look Into My Life
- Premium layer
- Future / not in launch

If an idea does not clearly fit one of those buckets yet, it is not ready for build.
